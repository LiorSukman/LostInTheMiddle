# Serial Position Effects in LLMs — Research Plan
### Multi-Document QA | Preamble Length · Relative vs Absolute Position · Von Restorff

---

## Why Multi-Document QA Over KV Retrieval

The original Liu et al. (2023) paper showed that the KV retrieval task produced inconsistent results: Claude models showed almost no positional effect at all, and the U-curve was much weaker at 8–16K token ranges. The multi-document QA task produced the cleaner, more replicable U-shaped curve — and it's also the more RAG-relevant setup. This plan uses QA throughout.

---

## Dataset: Wikipedia 2024-2025

### Background: TriviaQA was attempted and abandoned

We initially built a dataset from TriviaQA (Wikipedia config) following the Liu et al. approach. After filtering 1,600 questions through short-answer, distractor availability, and closed-book gates, we obtained 147 verified questions. However, GPT-4o-mini's parametric memory (knowledge cutoff) was either too strong — the model answered most questions correctly regardless of gold document position, or the questions were too ambiguous / outdated, producing a flat accuracy curve (~48-51% across all positions) with no detectable U-shape. The dataset was abandoned in favor of a custom dataset built from post-training-cutoff content.

### Current Dataset: Wikipedia 2024-2025 Events

To eliminate parametric memory contamination, we built a dataset from Wikipedia articles about events after GPT-4o-mini's training cutoff (October 2023). The model cannot have memorized these facts, so every correct answer must come from the provided documents.

**Pipeline (implemented in `DatasetBuilder/`):**

1. **Harvest** — Fetch 22 Wikipedia articles on recent events (Oscars, Olympics, Nobel Prizes, elections, Grammys, Wimbledon, Tour de France, Booker Prize, etc.)
2. **Chunk** — Split articles into 80-115 word passages (573 chunks total)
3. **QA extraction** — Generate question-answer pairs from chunks using Claude
4. **Filter** — Remove questions answerable from general knowledge (closed-book gate), verify answer presence in source passage (oracle check), reject low-quality items (long answers, tautologies)
5. **Distractor ranking** — For each question, rank all non-gold chunks by semantic similarity to the question (sentence-transformers `all-MiniLM-L6-v2`), select top-k as distractors
6. **Validation** — Verify answer presence in gold passage, check for distractor contamination (answer appearing in a distractor)

**Final datasets:**
- `final_dataset.json` — 196 questions, 19 distractors each (20-document prompts)
- `final_dataset_30.json` — 196 questions, 29 distractors each (30-document prompts)

**Domain distribution:** Film/Awards (66), Sports (53), Music (28), Politics (19), Literature (15), Science (15)

**Quality checks:** 0 answer presence failures, 0 distractor contamination, 0 duplicate questions.

## Prompt Template

```
{PREAMBLE}

Below are {N} documents. Use only the information in these documents to answer the question. If the answer is not in the documents, say "I don't know."

Document [1]: {doc_1_text}
Document [2]: {doc_2_text}
...
Document [N]: {doc_N_text}

Question: {question}
Answer with only the answer, no explanation:
```

The instruction "Answer with only the answer, no explanation" is critical for exact-match scoring.

---

## Scoring

```python
def normalize(text):
    text = text.lower().strip()
    text = re.sub(r'[^\w\s]', '', text)  # strip punctuation
    return text

def score(response: str, answer_aliases: list[str]) -> bool:
    norm_response = normalize(response)
    return any(normalize(alias) in norm_response for alias in answer_aliases)
```

Each question has multiple valid answer aliases (e.g., `['5', 'five']` or `["Conan O'Brien", "O'Brien"]`). Scoring uses case-insensitive substring containment against all aliases.

Manual validation pass: after each experiment, spot-check ~15 incorrect responses. Most errors are "I don't know" (genuine retrieval failures); occasional near-misses (correct answer phrased differently than any alias) are treated as noise since they occur symmetrically across conditions.

## Statistical Tests

All experiments use within-subjects designs (every question tested at every condition), so tests must account for paired/repeated observations.

### Primary tests

| Test | Purpose | When to use |
|---|---|---|
| **Cochran Q** | Omnibus test for whether accuracy differs across k related conditions (generalization of McNemar to k>2) | Test whether position or preamble condition affects accuracy across all levels |
| **McNemar** (with continuity correction) | Pairwise test for whether two related conditions differ in accuracy | Compare specific pairs (e.g., edge vs middle, none vs long preamble) |
| **Chi-squared test of independence** | Whether two categorical variables are associated | Quick check; less appropriate than Cochran Q for paired data but useful for confirmation |

### Correction for multiple comparisons

When performing multiple pairwise McNemar tests, apply **Bonferroni correction** (multiply raw p-values by number of comparisons, cap at 1.0).

### Effect size reporting

For each experiment, report:
- **U-effect**: edge accuracy − middle accuracy (the core LITM measure)
- **Per-position accuracy** with 95% CI (binomial standard error × 1.96)
- **Odds ratio** for edge vs non-edge conditions where relevant

### Position-sensitive question analysis

Questions that are always correct or always wrong regardless of position contribute zero signal to positional tests (they have no discordant pairs). We report the full-dataset analysis as primary, and include a supplementary "position-sensitive only" analysis that isolates the questions where position actually mattered. Note: Cochran Q values are identical for both (constant questions contribute nothing to the statistic), but chi-squared and descriptive statistics are cleaner on the filtered set.

## Experiment 0 — Replication Baseline

Replicate the Liu et al. (2023) U-shaped curve on our dataset.

### Design

**20-document version** (`01_replication_20doc.ipynb`):
- 196 questions, 20 documents each
- Gold document swept across positions 1, 5, 10, 15, 20
- Within-subjects: every question tested at every position
- 196 × 5 = 980 API calls

**30-document version** (`01_replication_30doc.ipynb`):
- 196 questions, 30 documents each
- Gold document swept across positions 1, 5, 10, 15, 20, 25, 30
- 196 × 7 = 1,372 API calls
- Rationale: more documents increase retrieval difficulty, pushing more questions into the position-sensitive zone

### Results

**20-document:** Marginal U-shape. Edge (pos 1, 20) = 80.4%, middle (pos 5, 10, 15) = 74.3%, U-effect = +6.0%. Cochran Q significant on position-sensitive subset only (p = 0.049).

**30-document:** Clear U-shape. Edge (pos 1, 30) = 80.9%, middle (pos 10, 15, 20) = 71.4%, U-effect = +9.4%.
- Cochran Q: Q = 27.28, df = 6, **p = 0.0001**
- McNemar pos 30 vs 20: **p = 0.0005**
- McNemar pos 1 vs 20: **p = 0.004**
- Position-sensitive questions (81/196): edge = 73.5%, middle = 50.6%, U-effect = +22.8%

**Gate: PASSED.** All subsequent experiments use 30-document prompts.

### Key design detail: distractor order

For the 30-doc dataset, each question has exactly 29 distractors. `random.sample(pool, 29)` returns all 29 in a shuffled order — every position gets the same set of distractors, only their arrangement differs. This ensures total prompt length and content are identical across positions for a given question, isolating position as the only variable.

## Experiment 1 — Preamble Length

### Hypothesis

A longer preamble before the document list shifts all documents to higher absolute token positions. If the model tracks absolute token position (RoPE decay hypothesis), primacy should weaken. If it reanchors to relative list position (structural hypothesis), the U-curve shape should be stable.

### Full design (ideal)

| Factor | Levels |
|---|---|
| Preamble | None (0 words), Short (~130 words), Medium (~550 words), Long (~1290 words) |
| Position | 1, 5, 10, 15, 20, 25, 30 |
| Questions | 196 (within-subjects) |
| **Total calls** | **196 × 7 × 4 = 5,488** |

### Reduced design (executed)

To manage cost and API limitations, we ran a reduced version:

| Factor | Levels | Rationale for reduction |
|---|---|---|
| Preamble | None, Short (~130w), Long (~1290w) | 3 levels still allow dose-response trend detection; medium is interpolatable |
| Position | 1, 5, 15, 25, 30 | Dropped 10, 20 — the middle trough was flat (70-72%) in Exp 0; position 15 suffices as trough reference |
| Questions | 196 (within-subjects) | No reduction |
| **Total calls** | **196 × 5 × 3 = 2,940** |

The reduced design preserves:
- Both edges (1, 30) for primacy/recency measurement
- Slope positions (5, 25) to detect asymmetric effects
- One true middle position (15) as trough reference
- Three preamble levels for trend detection and content-confound control

The full design can be extended later — saved results include question_id, position, and condition, so the checkpoint/resume logic will skip completed trials and only run new ones.

### Preamble source

Text from the "History of paper" Wikipedia article — verified to have zero topical overlap with any question domain (film, music, sports, politics, literature, science). The same fixed text is used for all questions at a given preamble level. The three levels are nested slices of the same article (short = first paragraph, long = first ~10 paragraphs), so the only variable across conditions is length, not content.

Stored in `data_misc/preambles.json` with all 4 levels (none, short, medium, long) for future extension.

### Key control: distractor order fixed per question

Distractors are shuffled once per question (seeded by question index), then the gold doc is inserted at each target position into that fixed sequence. This ensures:
- Across positions: same distractor order, gold doc just moves
- Across preamble conditions: identical document list, only preamble changes
- The only two independent variables are position and preamble length

### Results

**Finding: Preamble improves accuracy and eliminates the U-curve (opposite of RoPE decay prediction).**

| Condition | Overall | Edge (1,30) | Middle (15) | U-effect |
|---|---|---|---|---|
| No preamble | 77.6% | 82.9% | 73.5% | +9.4% |
| Short (~130w) | 83.4% | 85.7% | 81.6% | +4.1% |
| Long (~1290w) | 87.3% | 86.7% | 86.2% | +0.5% |

Statistical tests:
- Cochran Q (does preamble affect accuracy?): Q = 117.5, df = 2, **p < 0.000001**
- Cochran Q per condition (does position matter within each?):
  - No preamble: Q = 27.2, **p = 0.00002** (strong positional effect)
  - Short: Q = 9.4, p = 0.051 (positional effect disappearing)
  - Long: Q = 3.4, p = 0.494 (positional effect completely eliminated)
- Pairwise McNemar with Bonferroni (overall accuracy): none vs long, none vs short, short vs long

The dose-response gradient in the Cochran Q values (p = 0.00002 → 0.05 → 0.49) shows the preamble doesn't just improve accuracy — it specifically eliminates positional bias. The improvement is concentrated in middle positions (+12-17% at positions 5, 15, 25) while edges barely move (+3-4%).

### What to Plot
- 3 overlaid U-curves (one per preamble condition) — shows curve flattening
- Primacy (position 1) and recency (position 30) bar charts per condition
- These are implemented in `02_preamble_length.ipynb`

## Experiment 2 — Relative vs. Absolute Position (2×2 Factorial)

### Hypothesis

This experiment directly dissociates two competing accounts:

- **Absolute position hypothesis**: what matters is where the token sits in the full context (index from 0). Item at token 2000 is harder than the same item at token 50.
- **Relative position hypothesis**: what matters is where the document sits within the list (first, middle, last). The model reanchors regardless of upstream preamble.

> **Note:** Experiment 1 results already provide strong evidence against the absolute position (RoPE decay) hypothesis and toward the structural/relative hypothesis. Experiment 2 provides a cleaner, more direct test and adds the critical length-matched control (condition E).

### Design

Four conditions created by crossing preamble presence × target position in list:

| Condition | Preamble | Gold doc relative position | What it tests |
|---|---|---|---|
| A | None | Beginning (pos 1–2) | Baseline primacy |
| B | Long (~1500 tok) | Beginning (pos 1–2) | Same relative position, high absolute position |
| C | None | Middle (pos 13–17) | Baseline middle degradation |
| D | Long (~1500 tok) | Middle (pos 13–17) | Same relative middle, high absolute position |

Key comparisons:
- **A vs B**: both "first in list," but B has ~1290 words before the list starts. If accuracy drops from A→B, absolute position hurts even the first document.
- **C vs D**: both "middle of list," but D is additionally at high absolute token index. If C vs D differ significantly, absolute position is compounding the middle penalty.
- **A vs C**: the standard primacy vs middle comparison (replicates baseline)

> ⚠️ **Critical confound to control for:** Conditions B and D have longer total prompts than A and C. Add a 5th condition:
>
> **E**: Long preamble *appended after the document list* (before the question). This keeps total length matched to B/D but puts the filler in a different position. This also addresses the Experiment 1 finding — if the preamble's benefit comes from its position *before* the documents (not just total length), E should not show the same improvement as B/D.

### What to Plot
- 2×2 bar chart (primacy/middle × no preamble/preamble)
- Add condition E as a third preamble position to the bar chart

## Experiment 3 — Von Restorff Effect

### Core Idea

In human memory, a distinctive item in the middle of a list escapes the middle-suppression and is recalled at a higher rate than its position would predict. This is the Von Restorff (isolation) effect. No published work has tested whether a structurally or semantically distinctive document can be "rescued" from the lost-in-the-middle fate in LLMs.

This is the most novel experiment and the most directly applicable to RAG engineering.

### Design

**Fixed setup:**
- 30 documents per prompt
- Gold document always at middle positions (positions 12–18 of 30)
- 196 questions, within-subjects
- Baseline: no distinctiveness marker

**Three Von Restorff conditions (test each independently):**

---

#### Condition VR-1: XML Tag Wrapping

The gold document is wrapped in a distinctive XML tag:

```
Document [15]:
<key_document>
{gold_doc_text}
</key_document>
```

All other documents use plain `Document [N]: {text}` format.

**Why this is interesting for RAG:** This directly simulates a real-world RAG engineering practice — adding metadata markers to high-relevance chunks. If VR-1 significantly rescues accuracy, it's an immediately actionable finding.

> ⚠️ **Instruction-following confound:** Instruction-tuned models may treat XML-like tags as system-level signals and attend to them regardless of their position. To test this, run a control where you wrap a *distractor* document in the same XML tag while the gold document is at the same position without the tag. If the model now answers incorrectly (following the tag to the distractor), that's instruction-following, not Von Restorff.

---

#### Condition VR-2: Style/Language Shift

The gold document is rewritten or translated into a visually distinct style:

**Option 2a — Bullet points:** Convert the gold document's prose into bullet points while all distractors remain as prose paragraphs.

**Option 2b — Different language:** Translate the gold document to French (or another language) while distractors remain in English. The answer string should remain in English within the passage (e.g., names, dates).

> ⚠️ **Translation quality matters.** If your translation is low quality, the model may fail to extract the answer even if it attends to the document. Always manually verify that the answer string survived translation correctly.

---

#### Condition VR-3: Anomalous Framing

The gold document is preceded by a short anomalous or seemingly unrelated sentence:

**Option 3a — Unrelated opener:**
```
Document [15]: [Note: The following passage was retrieved from an unusual source.] {gold_doc_text}
```

**Option 3b — Confidence score:**
```
Document [15]: [Retrieved with high confidence score: 0.97] {gold_doc_text}
```

Option 3b is particularly interesting because it simulates a RAG system that passes retrieval scores as metadata into the prompt.

> ⚠️ **Confound with instruction following:** The control is to put a *low* confidence score on the gold document: `[Retrieved with confidence score: 0.12]`. If the model then ignores the gold document, that confirms it's following the score instruction, not the isolation effect.

---

### What to Measure Across All VR Conditions

- **Rescue rate**: `(accuracy_VR_condition - accuracy_baseline_middle) / (accuracy_baseline_edge - accuracy_baseline_middle)` — how much of the primacy advantage did distinctiveness recover?
- Accuracy at middle positions for each condition overlaid on baseline
- Separately report: cases where distinctiveness *hurt* accuracy

## What This Can and Cannot Claim

**Can claim:**
- Whether preamble length modulates U-curve shape (empirical finding: it eliminates the U-curve)
- Whether absolute vs. relative position drives the effect (Exp 1 suggests relative; Exp 2 provides direct test)
- Whether distinctiveness markers rescue middle documents from positional suppression (no prior published test)
- Which Von Restorff manipulation is most effective and most likely to generalize to real RAG systems
- That adding neutral preamble text before retrieved documents can improve RAG accuracy (actionable finding)

**Cannot claim:**
- Mechanism (observing behavior, not probing attention weights or activations)
- Generalization beyond GPT-4o-mini
- That the Von Restorff effect here is the same cognitive mechanism as in humans
- Causal claims about RoPE — we can only say the RoPE decay hypothesis's predictions were not supported

**Honest framing:** "Empirical study, N=196 questions, within-subjects design, single model (GPT-4o-mini), statistically significant results on key measures."
